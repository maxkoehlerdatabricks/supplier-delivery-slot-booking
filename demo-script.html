Duration: 45–60 min
Products: Lakebase + Apps

Part 1 The Business Problem 2 min

"Our LiDAR plant receives hundreds of deliveries per week. Suppliers need to book dock slots in advance so the warehouse team can plan. Today this is done by email and phone — chaotic, double-bookings, no visibility. We're going to build a modern solution using two new Databricks products: Lakebase for the transactional backend, and Databricks Apps for the hosted web application."

Part 2 Lakebase Deep Dive 25 min

2.1 — Project Overview

  1. Open the Databricks workspace sidebar → click the App Switcher (grid icon, top-left) → select Lakebase
  2. You see the list of Lakebase projects. Click delivery-slot-booking.
"Lakebase is a fully managed, PostgreSQL-compatible OLTP database built into Databricks. It gives us low-latency reads and writes for application workloads — something Delta Lake alone isn't designed for. Let me walk you through every capability."

2.2 — Project Settings

  1. Click Settings in the left sidebar.
  2. Point out:
    • Display name and Resource name (immutable project ID)
    • PostgreSQL version — we're on Postgres 17
    • Serverless usage policy — attribute compute costs to a budget policy
    • Custom tags — key-value pairs for cost tracking and organization
    • History window — currently set to 7 days. Controls how far back we can do point-in-time restore. Range is 2–30 days.
    • Maintenance window — automatic Postgres minor version upgrades and security patches happen here
    • Permissions — three levels: CAN CREATE, CAN USE, CAN MANAGE
"Everything is serverless — no VMs to manage, no patching to schedule manually. Lakebase handles Postgres upgrades automatically in the maintenance window you choose. The history window is key: it determines how far back you can restore or branch from."

2.3 — Branches (Git-Style Workflows)

  1. Click Branches in the left sidebar.
  2. Show the branches table: production (default, protected) and dev.
  3. Point out the parent relationship, compute status, data size, and last active time.
"Lakebase has a unique branching model inspired by Git. Each branch is an instant, isolated copy of your database using copy-on-write storage — no data duplication. You only pay for the data that changes on the branch."
  1. Click production → point out the shield icon (protected branch).
"The production branch is protected — it cannot be deleted, reset, or archived. This is your golden copy."
  1. Click dev branch → show that it was branched from production.
"The dev branch was created from production. Developers can run migrations, test schema changes, and load test data here without touching production."

Create a new branch (live)

  1. Click Create branch button.
  2. Fill in:
    • Name: feature-returns
    • Branch type: "Current data" (latest from production) — also show the "Past data" option and explain point-in-time branching
    • Expiration: 7 days
  3. Click Create.
"Branch creation is instant — even for terabytes of data. Under the hood, it's copy-on-write, so it takes zero extra storage initially. We can also branch from a specific point in time within our history window — that's how you'd recover from a bad migration."

Schema Diff

  1. Go back to the dev branch → click Schema diff.
  2. Show the side-by-side DDL comparison (green = added, red = removed).
"Schema diff lets you compare any child branch against its parent. It shows exactly what DDL changed — added tables, altered columns, dropped indexes — making code review for database changes possible."

Branch Operations

  1. Point out (don't execute): Reset from parent, Set as default, Delete branch.
"You can reset a branch to re-sync with the parent's latest data — like a git reset --hard. You can also set any branch as the default, which exempts it from concurrency limits."

2.4 — Compute (Autoscaling & Scale-to-Zero)

  1. Click the Computes tab on the production branch.
  2. Show the primary compute: autoscaling range (e.g., 8–16 CU), scale-to-zero enabled.
"Lakebase separates compute from storage. Each Compute Unit is 2 GB of RAM. Our production compute autoscales between 8 and 16 CU — the system adjusts dynamically based on workload without restarts or connection interruptions."
  1. Click Edit on the primary compute → show:
    • Min/Max CU sliders
    • Scale-to-zero toggle and inactivity timeout (currently 24 hours)
"Scale-to-zero means when there's no activity, compute suspends entirely — zero cost. When a new query arrives, it reactivates in a few hundred milliseconds. The inactivity timeout is configurable down to 60 seconds."

Read Replicas

  1. Click Add Read Replica.
  2. Show the configuration form (don't necessarily create one).
"You can add up to 6 read replicas per branch. Each replica shares the same storage layer — instant creation, no data copying. They have independent autoscaling and scale-to-zero settings. This is perfect for separating analytical read workloads from transactional writes."

2.5 — SQL Editor

  1. Click SQL Editor in the left sidebar.
  2. Select branch: production, database: delivery_app.
  3. Run: \dt to list all tables.
"The SQL Editor supports psql meta-commands. Here we see our OLTP tables: dock_slot and delivery_booking, plus the synced tables ekko and ekpo_enriched from the Lakehouse."
  1. Run a query:
SELECT b.booking_id, b.po_number, b.status, b.driver_name,
       ds.dock_id, ds.slot_date, ds.time_window_start
FROM delivery_booking b
JOIN dock_slot ds ON b.slot_id = ds.slot_id
WHERE b.status = 'confirmed'
ORDER BY ds.slot_date, ds.time_window_start
LIMIT 10;
  1. Show the result set. Click Explain to show the query plan.
"Explain shows the estimated execution plan — row counts, join strategies, index usage."
  1. Click Analyze to run EXPLAIN ANALYZE.
"Analyze actually executes the query and shows real timings and row counts — essential for performance tuning."
  1. Click the Export button → show JSON, CSV, XLSX options.
"Results can be exported directly to JSON, CSV, or Excel."
  1. Show multiple statement execution — run two queries separated by semicolons and point out the numbered result tabs.

Extensions

  1. Run:
SELECT * FROM pg_available_extensions
WHERE name IN ('vector', 'postgis', 'pg_stat_statements', 'pg_graphql')
ORDER BY name;
"Lakebase supports 46 PostgreSQL extensions including pgvector for AI embeddings, PostGIS for geospatial, pg_graphql for GraphQL, and pg_stat_statements for query performance analysis."

2.6 — Tables Editor

  1. Click Tables in the left sidebar.
  2. Navigate to delivery_app → public → dock_slot.
  3. Show the table grid with data.
"The Tables Editor gives you a spreadsheet-like view of your data."
  1. Inline editing: Click a cell (e.g., capacity), change the value, show the "Save x changes" button. (Revert it.)
  2. Add record: Click "Add record" — show DEFAULT fields auto-populated.
  3. Delete records: Check a few rows → show "Delete x records" button.
  4. Filters: Add a filter (e.g., dock_id = DOCK-A), show how it creates a saved View.
  5. Column toggle: Hide/show columns.
  6. Export: Click export → JSON, CSV, XLSX.

Database Studio

  1. Click Database studio (top navbar).
  2. Browse: schemas, tables, views, roles, and policies in a tree view.
"Database Studio gives you a single pane of glass over all database objects — schemas, tables, views, roles, and security policies."

2.7 — Synced Tables (Lakehouse → Lakebase)

  1. Navigate to Unity Catalog (switch back to main Databricks workspace).
  2. Find the ekko table in the catalog browser.
  3. Click on it → show the Overview tab with sync information.
"Synced Tables are how we bridge the Lakehouse and Lakebase. Our SAP purchase order data lives in Delta Lake. We've created a one-way sync that replicates it into Lakebase as read-only PostgreSQL tables."
  1. Point out:
    • Sync mode — Snapshot vs. Triggered vs. Continuous
    • Pipeline status — the underlying Lakeflow Declarative Pipeline
    • Last sync timestamp
    • "Sync now" button for manual trigger
"There are three sync modes: Snapshot does a full copy — best when >10% of rows change. Triggered does incremental updates on a schedule, using Change Data Feed. Continuous gives near-real-time sync with as low as 15-second intervals. Our demo uses Snapshot, but for production you'd likely use Triggered or Continuous."

2.8 — Lakehouse Sync (Lakebase → Lakehouse)

  1. Go back to Lakebase → select the production branch.
  2. Click the Lakehouse sync tab.
"This is the reverse direction — Lakebase to Lakehouse. It continuously replicates Lakebase Postgres tables into Unity Catalog as managed Delta tables using CDC (Change Data Capture). It uses the wal2delta Postgres extension under the hood."
  1. If enabled, show:
    • Overall sync status ("Syncing")
    • Sub-tabs for Schemas and Tables
    • Per-table: source name, destination table, status, Committed LSN, Last Sync timestamp
"Every insert, update, and delete in Lakebase is captured and written to Delta tables in SCD Type 2 format — you get full history. System columns like _pg_change_type, _pg_lsn, and _timestamp are added automatically. This means your BI dashboards and ML models can consume real-time transactional data without querying Lakebase directly."
  1. If not enabled, show the "Start sync" button and walk through the configuration:
    • Select source database and schema
    • Select target Unity Catalog catalog and schema
    • Mention the REPLICA IDENTITY FULL requirement

2.9 — Metrics Dashboard

  1. Click Metrics in the left sidebar.
  2. Select branch: production, compute: primary.
  3. Set time period to Last hour (or Last day if more data).
"The Metrics Dashboard gives you full observability with 19 metrics."
  1. Walk through the key charts:
    • Allocated vs Used CPU — shows autoscaling in action
    • Allocated vs Used vs Cached RAM — memory utilization
    • Active / Idle / Total / Max Connections — connection management
    • Database Size — logical data size
    • Rows Inserted / Updated / Deleted — write throughput
    • Deadlocks — should be zero in healthy systems
    • Local File Cache Hit Rate — how efficiently the cache serves reads
    • Working Set Size (5m, 15m, 1h) — how much data is actively accessed
    • Replication Delay (bytes and seconds) — relevant for read replicas
"Notice how CPU and RAM scale automatically. When the app is idle, you'll see metrics drop to zero — that's scale-to-zero in action. No cost when there's no workload."

2.10 — Connection & Credentials

  1. Click Connect (on the production branch).
  2. Show the Connect dialog:
    • Branch, database, and role dropdowns
    • Compute selector (primary or read replica)
    • Connection string — copy button
    • Pooler connection string (PgBouncer)
"Lakebase supports two auth methods: OAuth tokens for Databricks identities — these are short-lived (60 min) and auto-refreshed. And native Postgres password roles for traditional app authentication. The built-in PgBouncer connection pooler supports up to 10,000 concurrent connections in transaction mode — ideal for serverless apps and web APIs."
  1. Mention supported clients: psql, pgAdmin, DBeaver, and programmatic access via Python, JavaScript, Go, .NET.

2.11 — Data API (PostgREST)

  1. Click Data API in the left sidebar.
  2. Show the Enable/Configure screen.
"The Data API exposes your Lakebase tables as a RESTful API via PostgREST — no backend code needed. You get automatic CRUD endpoints with filtering, sorting, pagination, and joins via foreign keys. It auto-generates an OpenAPI 3.0 spec. You can configure exposed schemas, max rows, CORS settings, and row-level security for multi-tenant isolation."
  1. Point out configuration options: exposed schemas, max rows, CORS, OpenAPI endpoint.

2.12 — Security Features

"A few security features worth mentioning:

• Encryption at rest with optional customer-managed keys (CMK) via AWS KMS, Azure Key Vault, or Google Cloud KMS
• SSL required on all connections (sslmode=require)
• Inbound Private Link for keeping Postgres traffic off the public internet
• IP access lists for network-level restriction
• Row-Level Security via the Data API for multi-tenant data isolation
• PostgreSQL GRANT/REVOKE for fine-grained table-level permissions
• Role management in Database Studio — create OAuth roles for Databricks identities or native password roles"

2.13 — Instant Restore

"One last Lakebase feature: Instant Restore. Using our 7-day history window, we can restore the database to any point in time — or more practically, create a branch from any past moment. This is copy-on-write, so it's instant even at terabyte scale. It's how you'd recover from a bad migration or data corruption."

Part 3 Databricks Apps Deep Dive 20 min

3.1 — Apps Overview

  1. Navigate to Databricks Apps in the workspace sidebar (or App Switcher).
  2. Show the list of apps. Click delivery-slot-booking.
"Databricks Apps is a managed application hosting platform. You build your app in any framework — Python or Node.js — and Databricks handles deployment, authentication, networking, and compute. No Kubernetes, no cloud infrastructure to manage."

3.2 — Overview Tab

  1. On the app detail page, point out:
    • Status: RUNNING (green indicator)
    • App URL: https://delivery-slot-booking-<workspace-id>.<region>.databricksapps.com
    • Deployment source — workspace folder
    • Resources listed: SQL Warehouse
    • Service principal — auto-created, dedicated to this app
"Every app gets an auto-generated HTTPS URL with TLS 1.3. It also gets a dedicated service principal — this is the identity the app uses to access Databricks resources. The URL is immutable after creation."

3.3 — Resources

  1. Scroll to the Resources section.
  2. Show the configured SQL Warehouse resource.
"Resources are how Apps connect to Databricks services. There are 12 resource types: Lakebase databases, SQL Warehouses, Model Serving Endpoints, Genie Spaces, Secrets, Unity Catalog tables, Vector Search indexes, MLflow experiments, Lakeflow Jobs, other Apps, UC connections, and UDFs. Each resource gets a default key that can be referenced in app.yaml via valueFrom — credentials are injected automatically as environment variables. No secrets in code."
  1. Mention: "Our app also uses Lakebase directly via environment variables — the Postgres connection details are in app.yaml."

3.4 — Supported Frameworks

"Databricks Apps supports:

• Python: Streamlit, Dash, Gradio, Flask, FastAPI, Shiny
• Node.js: React, Angular, Svelte, Express
• Mixed: Python backend + Node.js frontend together

Our app uses FastAPI for the backend API and React + TypeScript + TailwindCSS for the frontend, built with Vite. The app.yaml runs uvicorn app:app --workers 4."

3.5 — app.yaml Configuration

  1. Open the app source code in the workspace (or show it on screen).
  2. Walk through app.yaml:
command:        # Custom startup command (uvicorn with 4 workers)
env:            # Environment variables — Lakebase connection params
resources:      # SQL warehouse with CAN_USE permission
"The app.yaml is entirely optional — defaults work for simple apps. But it lets you customize the startup command, define environment variables (either hardcoded or referencing resources via valueFrom), and declare resource dependencies. Never put secrets directly in value — use a Secret resource with valueFrom."

3.6 — Deployments Tab

  1. Click the Deployments tab.
  2. Show the deployment history — timestamps, status, source.
"Every deployment is versioned. You can see the full history here — when it was deployed, from what source, and its status. Redeploying after a code change doesn't require recreating the app."
  1. Mention deployment sources:
    • Workspace folder (what we're using)
    • Git repository (GitHub, GitLab, Bitbucket, Azure DevOps) — supports branches, tags, commit SHAs
    • Declarative Automation Bundles (DABs) — infrastructure-as-code in databricks.yml
    • GitHub Actions for CI/CD

3.7 — Compute Configuration

  1. Click the Settings tab → show the Compute section.
"Apps run on serverless infrastructure. Two sizes: Medium (2 vCPUs, 6 GB RAM, 0.5 DBU/hour) and Large (4 vCPUs, 12 GB RAM, 1 DBU/hour). You can change the size seamlessly — the app keeps running on the old size until the new one is ready. Stopped apps cost nothing."

3.8 — Authentication & Authorization

  1. Click the Authorization tab (if visible) or discuss:
"Apps support two identity models:

1. Service Principal identity — the app acts as itself. The DATABRICKS_CLIENT_ID and DATABRICKS_CLIENT_SECRET are auto-injected. This is the default.

2. On-Behalf-of-User (Public Preview) — the app acts with the logged-in user's identity and permissions. This enforces Unity Catalog policies including row-level filters and column masks. Users see an OAuth consent screen on first access.

Our app uses the service principal model for Lakebase access — the app authenticates with a Postgres password role."

3.9 — Permissions

  1. Click the Share button on the Overview tab.
  2. Show the permissions dialog:
    • CAN MANAGE — full control (settings, permissions, edit, delete)
    • CAN USE — can run and interact, cannot modify
"Sharing is simple: add users, groups, or service principals with either CAN MANAGE or CAN USE. You can also toggle organization-wide access — 'Anyone in my organization can use' grants CAN USE to all account users."

3.10 — Logs

  1. Click the Logs tab.
  2. Show the stdout/stderr output — FastAPI startup logs, request logs.
"Application logs are available here in real time — stdout and stderr from your app process. You can also access them by appending /logz to the app URL, or stream them via CLI with databricks apps logs. Note: logs are not persisted when compute shuts down."
  1. Mention the Telemetry feature:
"For persistent observability, you can enable App Telemetry (Beta) — it collects OpenTelemetry traces, logs, and metrics and persists them to Unity Catalog tables. You get otel_logs, otel_spans, and otel_metrics tables for long-term analysis and dashboarding."

3.11 — Insights Tab

  1. Click the Insights tab (if available, Beta).
"Insights tracks unique user access events, app health status, and service health indicators — useful for understanding adoption and uptime."

3.12 — Environment Tab

  1. Click the Environment tab.
  2. Show the runtime environment: installed packages, environment variables.
"The Environment tab shows exactly what's running — Python/Node.js version, installed packages, and all environment variables. Useful for debugging dependency issues."

3.13 — Tags

  1. Show the Tags section in the right panel.
"Tags let you organize and categorize apps with key-value pairs. Governed tags can indicate certification status or lifecycle stage — useful for enterprise app catalogs."

3.14 — Networking

"On the networking side:

• IP Access Lists restrict who can reach your app
• Front-End Private Connectivity routes traffic through your VPC instead of the public internet
• Network Connectivity Configurations give you stable public IPs for outbound traffic
• Network Policies (Enterprise tier) enforce egress restrictions — prevent data exfiltration by limiting which external domains the app can reach
• All traffic uses TLS 1.3, control plane uses mutual TLS, and ephemeral storage is AES-256 encrypted"

3.15 — Additional Capabilities

"A few more things worth knowing:

• Embedding: You can embed any Databricks App in an external website via iframe
• MCP Server hosting: Apps can host Model Context Protocol servers for AI agent integration
• State management: Apps don't preserve in-memory state across restarts — use Lakebase, SQL tables, or UC Volumes for persistence
• CI/CD: Full support for GitHub Actions, DABs, Terraform, REST API, and Python SDK
• Limits: 100 apps per workspace, 10 MB max file size, Medium or Large compute"

Part 4 The Application in Action 10 min

4.1 — Open the App

  1. Click the app URL from the Overview tab (or copy/paste it).
  2. The app opens in a new browser tab — dark "mercedes" themed UI.

4.2 — Supplier Portal (Book a Slot)

  1. You're on the Supplier Portal page (/).
  2. Step 1 — Select Date: The calendar shows available dates fetched from Lakebase (GET /api/slots/dates). Pick a date.
  3. Step 2 — Choose Slot: A grid appears showing dock IDs (DOCK-A through DOCK-E) vs time windows (06:00–08:00 through 20:00–22:00). Green = available, Grey = full. Click an available slot.
  4. Step 3 — Select PO: A dropdown shows Purchase Orders fetched from the synced ekko table in Lakebase. Select a PO number.
  5. Step 4 — Confirm: Enter truck plate and driver name → Submit.
"Notice the data flow: slot availability comes from the dock_slot table in Lakebase (OLTP). The PO dropdown comes from the ekko table that was synced from Delta Lake into Lakebase. The booking INSERT goes back to Lakebase. All sub-millisecond latency — this is why you need an OLTP database for application workloads."

4.3 — Warehouse Clerk (Manage Bookings)

  1. Click Warehouse Clerk in the nav bar (/clerk).
  2. Select a PO from the dropdown.
  3. Show the PO Header — vendor, PO type, date, purchasing org (from synced ekko).
  4. Show the Line Items table — LiDAR materials, quantities, prices (from synced ekpo_enriched).
  5. Below, show Delivery Bookings linked to this PO.
  6. Click through the status workflow: Requested → Confirmed → Checked In → Completed.
  7. Show the Cancel action on a booking.
"The clerk sees a unified view: SAP master data from the Lakehouse (via synced tables) and transactional booking data from Lakebase — all served by the same PostgreSQL connection. Status transitions are enforced by the API with validation logic."

4.4 — Dashboard

  1. Click Dashboard in the nav bar (/dashboard).
  2. Show:
    • Stat cards — total bookings by status (requested, confirmed, checked_in, completed, cancelled)
    • Dock Utilization — progress bars per dock with green/yellow/red coloring
    • Recent Activity — last 10 bookings table
  3. Wait 30 seconds to see the auto-refresh.
"The dashboard aggregates data from a single summary API call that runs several PostgreSQL queries. It auto-refreshes every 30 seconds. This is the kind of real-time operational view that's only possible with an OLTP database — you wouldn't run these queries against Delta Lake on every page load."

Part 5 Architecture Recap 3 min

┌─────────────────────┐     Sync (Delta → PG)     ┌───────────────────────┐
│   Unity Catalog     │ ─────────────────────────→ │      Lakebase         │
│   (Delta Lake)      │                            │   (PostgreSQL 17)     │
│                     │ ←───────────────────────── │                       │
│  • ekko             │     Lakehouse Sync         │  • dock_slot          │
│  • ekpo_enriched    │     (PG → Delta CDC)       │  • delivery_booking   │
│  • dock_slot (gold) │                            │  • ekko (synced, RO)  │
│  • delivery_booking │                            │  • ekpo_enriched (RO) │
└─────────────────────┘                            └───────────┬───────────┘
                                                               │
                                                         asyncpg (SSL)
                                                               │
                                                   ┌───────────┴───────────┐
                                                   │   Databricks App      │
                                                   │   FastAPI + React     │
                                                   │   (Serverless)        │
                                                   └───────────┬───────────┘
                                                               │
                                                          HTTPS/TLS 1.3
                                                               │
                                                   ┌───────────┴───────────┐
                                                   │   Users (Browser)     │
                                                   │  • Suppliers          │
                                                   │  • Warehouse Clerks   │
                                                   │  • Plant Managers     │
                                                   └───────────────────────┘
"To summarize: Lakebase gives us a fully managed PostgreSQL database with autoscaling, scale-to-zero, instant branching, synced tables from the Lakehouse, Lakehouse Sync back to Delta, 46 extensions, a built-in Data API, connection pooling, metrics, and enterprise security — all serverless. Databricks Apps gives us managed hosting for any Python or Node.js web framework with built-in authentication, resource management, telemetry, and zero infrastructure to manage. Together, they turn Databricks into a full-stack application platform — from data lake to production app in a single control plane."
